01CLAUDE.md — 把规则写进文件，一劳永逸

每次 session 自动生效，不用反复交代

它是什么？

Claude Code 每次启动先读项目根目录的 CLAUDE.md。你把「这个项目的规矩」写在里面，Claude 永远记得——不用每次 session 重新说一遍。本质是一个持久化的 system prompt。

/init 一键生成

$ /init
# Claude 自动扫描项目生成 CLAUDE.md

该写什么？

分类典型内容重要度
常用命令npm run test / python -m pytest必写
代码风格TypeScript strict / PEP8 / 命名规范必写
目录约定数据在 /data/raw，模型存 /models/必写
禁止操作不改 /legacy / 不直接 push main推荐
工作流PR 前跑 lint，commit 格式要求可选
💡 Tips

越短越有效 — 超过 150 行就等于没写。Claude 自己能判断的别写，只写它推断不出来的。能做成 hook 的不要放这里。

02先规划后执行，开两个 session

边想边做是最大的浪费

为什么要分开 session？

规划时的「讨论噪音」会污染执行阶段的 context。新 session + SPEC.md = Claude 专注执行，不被之前的来回干扰。

  1. 一句话描述 — 你先说需求
  2. 访谈 Claude — 问你问题
  3. SPEC.md — 写下来确认
  4. 新 session — 干净执行
  5. 验证 — 跑测试截图
💡 让 Claude 问你

开头只给一句话，然后说「你来问我问题」。Claude 会把你没想到的坑提前暴露。

访谈维度 — Claude 应该问的

维度问题
技术选型用哪个库？有没有可复用的现有代码？
成功标准怎样算做完？测试怎么写？
边界情况空数据？并发？权限？报错怎么展示？
下游影响改了这里会不会破坏其他模块？
可还原性出错了怎么回滚？数据会不会丢？

03Session 管理 — 上下文是稀缺资源

用完就清，别让垃圾堆积

核心原则：一个 session 只做一件事

把不相关任务混在一个 session 里，Claude 的注意力被稀释，输出变差。像浏览器标签页，一个任务一个 session。

关键技巧

  1. 两次纠错失败 → 立刻 /clear
    继续纠正只会越来越乱。清空重来，把教训写进初始 prompt。
  2. /compact：长 session 续命
    快到 context 上限时用 /compact，Claude 自动摘要历史，释放约 60% 空间继续干。
  3. 重要决策写文件持久化
    跨 session 要记住的东西别靠记忆——写进 DECISIONS.md，下次 session 开头读一遍。
  4. 每次开头给 Claude 一个锚点
    把 SPEC.md / 报错日志 / 关键文件路径粘到第一条消息，Claude 立刻知道在做什么。

04没有验证，别说做完了

Claude 看上去没问题 != 真的没问题

先给成功标准，再让 Claude 动手

没有测试时 Claude 在猜你要什么。有了测试，它有明确目标，能自主迭代到通过。Test-first 不是仪式，是给 Claude 安装了一个目标函数。

应该给 Claude 的

  • 提前给 Claude 测试脚本 / pytest
  • 截图展示预期效果
  • 明确的成功 / 失败标准
  • 需要覆盖的 edge case
  • 容易踩的坑

常见错误

  • 「看上去对了」就合并
  • 只测 happy path
  • 让 Claude 自己定义完成
  • 跳过边界情况上线

05问题越具体，答案越有用

范围不限 = token 黑洞

「帮我看看这个 bug」是最贵的 prompt。不限范围让 Claude 读几百个文件、填满 context，最终给你一个表浅结论。把范围收到文件级，答案质量翻倍，token 消耗大幅下降。

太宽的问法正确做法
"帮我找这个 bug""在 auth/jwt.py 第 47 行找 token 过期的原因"
"分析整个项目性能""分析 db.py 里查询超 500ms 的函数"
"重构这个项目""把 Table 组件拆成 headless + styled 两层"
"理解这个报错""读 error.log 前 30 行，找 CORS 冲突根因"
💡 用 subagent 做大任务

复杂任务先拆成独立子任务，每个子任务单开 session，主 context 不被污染。

06MCP — 让 Claude 直接操作外部服务

不只是写代码，而是真的执行

MCP 是什么？

Model Context Protocol。连接后 Claude Code 可以直接查数据库、创 issue、读设计稿——不是帮你写代码，是它自己动手做。

添加 MCP Server

$ claude mcp add local-server /path/to/server
$ claude mcp add --transport sse remote https://api.example.com/sse
$ claude mcp list

MCP Server 能力一览

MCP ServerClaude 能直接做什么
GitHub读 PR diff → 写 review → 合并通过 CI 的 PR
PostgreSQL / SQLite查 schema → 生成 migration → 验证数据一致性
Notion / Linear从 issue 读需求 → 实现完自动关闭 ticket
Sentry / Datadog拉线上报错 → 定位源码 → 自动 patch
HuggingFace Hub查模型 metadata → 下载 checkpoint → 更新 model card

07Hooks — 不信任记忆，用自动化兜底

CLAUDE.md 是建议，Hooks 是法律

CLAUDE.md vs Hooks

CLAUDE.md 指令Hooks
执行率Claude 会尽量遵守100% 执行，无例外
复杂任务可能忘记独立于 Claude 推理之外
适合场景风格 / 偏好 / 约定必须做的操作
语气「最好这样做」「必须这样做」

Hook 类型

  1. PostToolUse：写完自动格式化
    Claude 每次改完代码，自动跑 black / prettier。不用担心它忘记。
  2. PreToolUse：危险命令拦截
    执行 bash 前检查是否含 rm -rf / sudo 等危险操作，拦截并提示确认。
  3. Stop：完成前验证
    任务结束前检查测试是否全过，没过则阻止 Claude 报告完成。
  4. Notification：操作审计日志
    把所有 Claude 操作写入 audit.log，方便复查和合规。

Hook 配置示例

// settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{"type": "command",
        "command": "black . && isort ."}]
    }]
  }
}

⑤ 给未来 AP：管理课程代码库

  1. CLAUDE.md 定义课程代码规范 — 注释语言、数据集路径、变量命名——一次写好，维护成本降到零。
  2. Hook 自动生成学生版 — 教师版改完，PostToolUse Hook 自动删答案生成学生练习版。
  3. gh CLI 批量处理学生提交 — Claude 写脚本 clone 所有 fork → 跑测试 → 生成反馈报告。
  4. 每门课独立 repo + CLAUDE.md — Context 隔离，不同课程互不干扰，每次 session 定位精准。
