关于本文档

这是 Anthropic 官方工程博客文章《Claude Code: Best practices for agentic coding》的中文翻译，发布于 2025 年 4 月 18 日，内容忠实原文并补充了 CLI 实战命令案例。

1定制你的开发环境

通过环境调优让 Claude 更快、更准、更省 token

Claude Code 是一个会自动将上下文拉入 prompt 的 agentic 编程助手。这种上下文收集消耗时间和 token，但可以通过环境调优来优化。

a. 创建 CLAUDE.md 文件

CLAUDE.md 是 Claude 在每次对话开始时自动拉入上下文的特殊文件。适合放以下内容：

适合写入 CLAUDE.md 的内容说明
常用 bash 命令构建、测试、类型检查等命令，让 Claude 随时知道怎么跑项目
核心文件和工具函数关键文件路径、常用工具的位置
代码风格规范ES modules vs CommonJS、命名规范、格式要求等
测试说明如何跑测试、跑哪些测试、测试框架约定
仓库规范分支命名、merge vs rebase、PR 要求等
开发环境pyenv 版本、编译器要求、环境变量说明
项目特有的坑已知的奇怪行为、警告、需要注意的地方

没有固定格式要求，保持简洁可读即可。

CLAUDE.md 示例

# Bash 命令
- npm run build: 构建项目
- npm run typecheck: 运行类型检查

# 代码风格
- 使用 ES modules (import/export)，不用 CommonJS (require)
- 尽量解构导入，如 import { foo } from 'bar'

# 工作流
- 改完一批代码后记得跑类型检查
- 优先跑单个测试，不要每次跑全套（太慢）

CLAUDE.md 文件位置

文件位置作用范围
项目根目录 CLAUDE.md最常用。提交进 git 可跨 session 共享给团队
项目根目录 CLAUDE.local.md本地私有，加入 .gitignore，不共享给他人
父目录 CLAUDE.md适合 monorepo：运行时自动读取所有父级文件
子目录 CLAUDE.md按需读取：处理子目录文件时自动拉入
~/.claude/CLAUDE.md全局生效：对所有 claude session 都有效
💡 用 /init 自动生成初稿

运行 /init，Claude 会自动扫描项目生成初始 CLAUDE.md。编码过程中按 # 键可以给 Claude 发指令，Claude 会自动写入对应的 CLAUDE.md。

b. 调优你的 CLAUDE.md

CLAUDE.md 会成为 Claude prompt 的一部分，所以应该像调优常用 prompt 一样去打磨它。常见错误是堆砌大量内容却不验证效果。要持续实验，找出真正有效的指令。

⚠️ 越精简越有效

如果 Claude 不看 CLAUDE.md 就能做对，就删掉那条指令，或转成 hook。重要规则加 IMPORTANT 或 YOU MUST 能显著提高遵守率。

c. 管理 Claude 的工具许可清单

Claude Code 默认对所有可能修改系统的操作都会询问权限（文件写入、bash 命令、MCP 工具等）。你可以把安全的、或容易撤销的操作加入许可清单，避免重复确认。

许可管理方式使用场景
会话中选择 Always allow临时快速授权
/permissions 命令交互式添加/删除，如 Edit、Bash(git commit:*)
编辑 .claude/settings.json持久化项目级许可，推荐提交进 git 共享给团队
编辑 ~/.claude.json持久化个人全局许可
--allowedTools CLI 参数针对单次 session 的临时许可

CLI 案例：预先授权常用操作

# 启动时预先允许文件编辑和 git commit
$ claude --allowedTools Edit,Bash(git commit:*)

# 在 session 内用 /permissions 添加
> /permissions add Edit
> /permissions add Bash(git commit:*)
> /permissions add mcp__puppeteer__puppeteer_navigate

d. 安装 gh CLI（GitHub 用户必装）

Claude 知道如何使用 gh CLI 与 GitHub 交互——创建 issue、开 PR、读评论等。没有 gh 时 Claude 也能用 GitHub API 或 MCP server，但 gh 最顺手。

# macOS
$ brew install gh
$ gh auth login

# Ubuntu/Debian
$ sudo apt install gh
$ gh auth login

2给 Claude 更多工具

bash 工具、MCP、自定义斜杠命令

a. 配合 bash 工具使用

Claude Code 继承你的 bash 环境，能访问所有已安装工具。但 Claude 不会自动知道你的自定义脚本，需要你明确告知：

告知方式示例
直接在 prompt 里说工具名和用法"用 my-deploy 脚本部署，my-deploy --help 看参数"
让 Claude 自己跑 --help"运行 my-tool --help 了解用法，然后帮我执行"
写入 CLAUDE.md把高频工具的名称和用法写进 CLAUDE.md 永久记录

CLI 案例：管道传入数据

# 把日志直接喂给 Claude 分析
$ cat error.log | claude '找出所有 500 错误，告诉我根因'

# 把 CSV 数据传入做清洗
$ cat survey_raw.csv | claude '找出 speeders 和 straight-liners，输出清洗后的版本'

# 让 Claude 读脚本 help 再执行
$ claude '先运行 ./scripts/deploy.sh --help，然后帮我部署到 staging'

b. 配合 MCP 使用

Claude Code 既是 MCP 服务端也是客户端。作为客户端可连接任意 MCP server。

配置层级适用场景文件位置
项目级 config在该目录运行 Claude 时生效.claude/settings.json
全局 config所有项目都生效~/.claude.json
提交进 .mcp.json整个团队开箱即用项目根目录 .mcp.json

CLI 案例：添加和调试 MCP server

# 添加本地 MCP server
$ claude mcp add my-server /path/to/server

# 添加远程 SSE MCP server
$ claude mcp add --transport sse remote https://api.example.com/sse

# 启动时开启 MCP debug 模式（排查配置问题用）
$ claude --mcp-debug

.mcp.json 示例：团队共享 Puppeteer + Sentry

{
  "mcpServers": {
    "puppeteer": { "command": "npx", "args": ["@mcp/puppeteer"] },
    "sentry": { "command": "npx", "args": ["@mcp/sentry"] }
  }
}

c. 自定义斜杠命令

把重复用的工作流（debug 循环、日志分析等）存为 Markdown 文件，放在 .claude/commands/ 目录，就能通过 / 菜单调用。支持 $ARGUMENTS 传参。提交进 git 可以共享给整个团队。

.claude/commands/fix-github-issue.md 示例

# 分析并修复 GitHub issue: $ARGUMENTS

步骤：
1. 用 `gh issue view $ARGUMENTS` 获取 issue 详情
2. 理解问题描述
3. 搜索代码库找到相关文件
4. 实现必要的改动修复问题
5. 编写并运行测试验证修复
6. 确保代码通过 lint 和类型检查
7. 写一个清晰的 commit message
8. push 并创建 PR

所有 GitHub 相关操作统一用 gh CLI。

CLI 案例：使用自定义斜杠命令

# 调用：修复 issue #1234
> /project:fix-github-issue 1234

# 个人命令（放 ~/.claude/commands/，所有 session 可用）
# 例：~/.claude/commands/daily-review.md
> /user:daily-review

3常见工作流

这些模式在各类代码库和语言中被证明有效

a. 探索 → 规划 → 编码 → 提交

适用于大多数问题的通用工作流：

步骤操作关键提示
① 探索让 Claude 读相关文件/图片/URL明确告诉它：先不要写代码
② 规划让 Claude 制定解决方案计划用 think / think hard / ultrathink 触发深度思考
③ 编码让 Claude 实现方案让它边实现边验证合理性
④ 提交让 Claude 提交并创建 PR可顺便让它更新 README / changelog
⚠️ 步骤 ① ② 是关键

跳过就会直接乱写代码。不先探索和规划，Claude 倾向于直接开始写代码。对需要深度思考的问题，先研究和规划能显著提升输出质量。

CLI 案例：探索-规划-编码流程

# 步骤 1：让 Claude 先只读代码，不写
> 读一下 auth/ 目录下处理 JWT 的文件，先不要写任何代码

# 步骤 2：触发深度规划
> think hard：给我一个把 JWT 换成 OAuth2 的迁移方案

# 步骤 3：确认计划后再执行
> 计划看起来没问题，开始实现，边做边验证每一步的合理性

# 步骤 4：提交并建 PR
> 提交改动，创建 PR，更新 CHANGELOG.md

b. TDD 工作流：先写测试，再写代码

Anthropic 内部最喜欢的工作流，适用于能用单测/集成测试/端到端测试验证的改动。TDD + agentic 编程威力倍增：

步骤操作注意
①让 Claude 根据预期输入/输出写测试明确说是 TDD，避免它写 mock 实现
②让 Claude 跑测试，确认测试会失败明确禁止此阶段写任何实现代码
③满意后让 Claude 提交测试单独提交测试
④让 Claude 写代码让测试通过，不允许改测试让它持续循环直到全部通过
⑤满意后提交实现代码可让子 agent 验证没有过拟合测试

c. UI 工作流：写代码 → 截图 → 迭代

类似 TDD，但用视觉截图替代测试作为验证标准：

# 给 Claude 设计稿 + 截图能力（Puppeteer MCP）
> 这是设计稿 [拖入图片]，用 Puppeteer 截图对比结果，
  迭代直到和设计稿一致再提交

# 没有 Puppeteer 时手动截图
> 实现这个组件，我会把截图粘给你，你来对比迭代

# 结果不够好时明确要求迭代
> 间距和设计稿不一致，再调整两到三轮
💡 Claude 的输出经过 2-3 轮迭代通常会好很多

第一版可能还不错，但给 Claude 看到结果（截图/测试输出/报错）让它自己看、自己调，2-3 轮后质量会显著提升。

d. Safe YOLO 模式（跳过所有权限确认）

用 --dangerously-skip-permissions 跳过所有权限询问，让 Claude 不中断地跑完。适合修 lint 错误、生成样板代码等低风险场景。

⚠️ 高风险！建议只在 Docker 容器内用

跳过权限可能导致数据丢失、系统损坏，或通过 prompt injection 泄露数据。生产环境务必在无外网的容器里运行。参考 Docker Dev Containers 实现参考。

# 在 Docker 容器内使用
$ docker run --rm -v $(pwd):/app my-claude-sandbox \
  claude --dangerously-skip-permissions \
  -p '修复所有 ESLint 错误，跑完测试确认没有破坏任何东西'

e. 用 Claude 探索和理解代码库

入职新项目时，用 Claude Code 做探索和学习。可以问和问同事一样的问题，Claude 会自动搜索代码库找答案：

> logging 是怎么实现的？
> 怎么新增一个 API endpoint？
> foo.rs 第 134 行的 async move { ... } 是干什么的？
> CustomerOnboardingFlowImpl 处理了哪些 edge case？
> 为什么第 333 行调的是 foo() 而不是 bar()？
> baz.py 第 334 行的 Java 等价写法是什么？

Anthropic 内部已把这个作为核心入职工作流，显著缩短新人 ramp-up 时间，减少对其他工程师的打扰。不需要特殊的 prompt，直接问就好。

f. 用 Claude 操作 git

Anthropic 工程师 90%+ 的 git 操作都用 Claude 完成：

git 操作怎么用 Claude
搜索历史"v1.2.3 版本包含了哪些改动？"
让 Claude 搜 git log写 commit message
"帮我写 commit message"Claude 自动看改动和历史
复杂 git 操作回滚文件、解决 rebase 冲突、比较和移植 patch

g. 用 Claude 操作 GitHub

Claude Code 可以处理大量 GitHub 日常操作，不用再背 gh 命令语法：

# 创建 PR（Claude 自动根据 diff 生成标题和描述）
> pr

# 一键修复 code review 评论
> 修复 PR #456 上所有 review 评论，改完 push 回同一分支

# 修 CI 失败
> 看一下 PR #456 的 CI 失败日志，找到原因并修复

# 批量处理 issues
> 遍历所有 open issues，按照 bug/feature/question 打标签

h. 配合 Jupyter Notebook 使用

Anthropic 的研究员和数据科学家用 Claude Code 读写 Jupyter notebook。Claude 能解读输出内容包括图片，是探索数据的快速通道。推荐工作方式：Claude Code 和 .ipynb 文件在 VS Code 中并排打开。

# 提交给同事前清理 notebook
> 帮我清理这个 notebook，删掉多余的 cell，
  让数据可视化在视觉上更美观

# 让图表更好看
> 把这些 matplotlib 图表改得 aesthetically pleasing，
  优化颜色、字体大小、标题格式

4优化你的工作流

适用于所有工作流的通用建议

a. 指令越具体，结果越好

具体的指令能显著提升 Claude Code 的成功率，尤其是第一次尝试。前期说清楚，后期少纠偏。

写法不佳改进写法
add tests for foo.pyfoo.py 里处理用户输入的 edge case 写测试，用 mock 隔离数据库
why does ExecutionFactory have such a weird api?ExecutionFactory 的 API 设计历史上有什么考量？用 git log 找最初设计
add a calendar widget在 HotDogWidget.php 第 47 行下面加一个日历 widget
💡 Claude 能推断意图，但不会读心术

具体 = 更符合预期。

b. 给 Claude 图片

Claude 能处理截图和设计稿，三种方式传入：

传入方式操作
截图粘贴macOS: Cmd+Ctrl+Shift+4 截图到剪贴板，Ctrl+V 粘贴（注意不是 Cmd+V）
拖拽图片直接拖入 prompt 输入框
文件路径在 prompt 里写图片文件路径
💡 没有图片时也要明确说明视觉要求

即使不传截图，告诉 Claude「结果要好看」也有帮助。对 UI 工作，视觉 mock 作为参照点能大幅提升质量。

c. 指定文件和 URL

用 Tab 键补全文件路径，快速引用任意目录下的文件。粘贴 URL 让 Claude 直接获取并阅读。对于频繁访问的域名（如 docs.foo.com），用 /permissions 加入许可清单避免重复授权。

# 指定文件（Tab 补全）
> 看一下 src/auth/[Tab]token.py，找出验证逻辑的问题

# 直接给 URL
> 读一下 https://docs.stripe.com/api/charges 然后帮我实现支付接口

# 添加域名到许可清单（不再每次询问）
> /permissions add WebFetch(docs.stripe.com)

d. 早纠偏、常纠偏

auto-accept 模式（Shift+Tab 切换）让 Claude 自主工作，但主动协作通常能得到更好的结果。

四个纠偏工具：

工具操作适用场景
先要计划让 Claude 先做计划，确认后再编码防止跑偏方向
Escape中断当前操作（思考/工具调用/文件编辑），重新引导发现方向错了
双击 Escape回到历史，编辑之前的 prompt想走另一条路
让 Claude 撤销撤销改动，通常配合双击 Escape实现方式完全不对

e. 用 /clear 保持 context 聚焦

长 session 中 context 会被无关对话、文件内容和命令填满，影响性能甚至干扰 Claude。任务间频繁使用 /clear 重置 context。

# 完成一个任务后清空 context 再开始下一个
> /clear

# context 快满时压缩（保留核心状态继续工作）
> /compact

# 重要决策持久化——不靠记忆靠文件
> 把我们刚才讨论的架构决策写进 DECISIONS.md

f. 用清单和草稿本处理复杂工作流

对于大型任务（代码迁移、批量修 lint、复杂构建脚本），让 Claude 用 Markdown 文件作为清单和工作草稿本：

# 步骤 1：让 Claude 列出所有错误并生成清单
> 跑 npm run lint，把所有错误（含文件名和行号）写入 lint-todo.md，
  格式：[ ] filename:line error_message

# 步骤 2：逐一修复并勾选
> 按照 lint-todo.md 的清单逐条修复，
  每修完一条在清单上打勾，再修下一条

g. 向 Claude 传入数据

传入方式命令示例最适合
直接粘贴粘入 prompt小段数据 / 报错
管道传入cat foo.txt | claude日志/CSV/大文件
bash 命令拉取"跑 git log -20 然后分析"git 历史/命令输出
MCP 工具拉取通过 MCP 拿数据库 schema实时外部数据
让 Claude 读文件"读 /src/config.py 再动手"理解项目结构
让 Claude 抓 URL"读 https://... 然后实现"API 文档/页面内容

大多数 session 会混合使用多种方式。例如：管道传入日志，再让 Claude 用工具拉取更多上下文来 debug。

5Headless 模式自动化你的基础设施

CI、pre-commit hook、构建脚本、自动化流水线

Claude Code 的 headless 模式用于非交互场景。用 -p 参数传入 prompt 启用，--output-format stream-json 获取流式 JSON 输出。

⚠️ 注意

headless 模式不跨 session 保持状态，每次 trigger 都是新 session。

a. 用 Claude 做 issue 分类

Headless 模式可以由 GitHub 事件触发的自动化调用。例如 Claude Code 官方仓库用 Claude 在新 issue 创建时自动打标签。

CLI 案例：issue 自动分类（GitHub Actions）

# .github/workflows/triage.yml
on:
  issues:
    types: [opened]

jobs:
  triage:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Triage with Claude
        run: |
          gh issue view ${{ github.event.issue.number }} | \
          claude -p "分析这个 issue，用 gh 打上合适的 label：
            bug/feature/question/docs，并回复一句说明" \
          --output-format stream-json
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

b. 用 Claude 做代码 lint 审查

Claude Code 能做传统 linting 工具检测不到的主观代码审查：typo、过期注释、误导性函数名/变量名等。

CLI 案例：pre-commit hook + 自动文档

# .git/hooks/pre-commit
#!/bin/bash

# 1. 让 Claude 做主观代码审查
git diff --cached | claude -p \
  "审查这次改动：找 typo、过期注释、误导性命名。
   有问题就列出来阻止提交" \
  --output-format stream-json

# 2. 给新函数自动加 JSDoc（headless 模式）
STAGED=$(git diff --cached --name-only --diff-filter=A)
if [ -n "$STAGED" ]; then
  claude -p "给以下文件中所有新函数添加 JSDoc 注释: $STAGED"
fi

6多 Claude 并行工作流

最强大的进阶用法——多实例协作

a. 一个写代码，另一个审查

让一个 Claude 写代码，另一个独立 context 的 Claude 来 review 或测试。类似多工程师协作，分离 context 往往能得到更好的结果。

CLI 案例：写代码+审查流水线

# 终端 1：Claude 写代码
$ claude
> 实现 UserAuth 模块的 OAuth2 登录功能

# 终端 2：另一个 Claude 独立审查
$ claude

b. 多个仓库 checkout 并行任务

Anthropic 内部很多工程师的做法：

步骤操作
①在不同文件夹创建 3-4 个 git checkout
②在不同终端标签页分别打开每个文件夹
③每个文件夹启动 Claude，分配不同任务
④轮流检查进度，批准/拒绝权限请求

c. 使用 git worktree（更轻量的并行方式）

Git worktree 是多 checkout 的轻量替代方案。每个 worktree 有独立工作目录，但共享 git 历史和 reflog。适合多个互不依赖的任务并行执行。

CLI 案例：git worktree 并行开发

# 创建 worktree 跑两个独立任务
$ git worktree add ../project-oauth feature/oauth
$ git worktree add ../project-dataviz feature/dataviz

# 终端 1：在 worktree 1 里启动 Claude
$ cd ../project-oauth && claude
> 重构认证系统，换成 OAuth2

# 终端 2：在 worktree 2 里启动另一个 Claude
$ cd ../project-dataviz && claude
> 实现数据可视化仪表板组件

# 两个任务完全独立，互不干扰

# 完成后清理
$ git worktree remove ../project-oauth
$ git worktree remove ../project-dataviz

③ CI/CD 自动化脚本

GitHub Actions 全自动 PR 工作流

# .github/workflows/ai-pr-review.yml
name: AI PR Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code
      - name: AI Code Review
        run: |
          git diff origin/main...HEAD | \
          claude -p "这是一个 PR 的 diff。请做代码审查：
            1) 找 bug 和安全问题
            2) 检查测试覆盖
            3) 评估代码可读性
           以 markdown 格式输出" \
          --output-format json > review.json

          # 把审查结果发到 PR 评论
          cat review.json | jq -r '.result' | \
          gh pr comment ${{ github.event.pull_request.number }} --body-file -
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
