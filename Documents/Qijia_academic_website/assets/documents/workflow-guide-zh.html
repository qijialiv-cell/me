第一章为何创建此工作流

1.1 问题所在

如果你曾做过严肃的学术工作——制作讲座幻灯片、起草研究论文、跑数据分析流水线——你一定深有体会：

  • 会话间记忆丢失。你继续上次的工作，但 Claude 不记得你当时为什么选择那个符号、指导老师批准了什么、上次修了哪些 bug。
  • 质量参差不齐。这张幻灯片间距完美；下一张内容溢出。这张回归表格格式标准；下一张缺少标准误。引用在 Overleaf 能编译，本地却报错。
  • 审阅全靠手动，精疲力竭。你亲手校对 140 张幻灯片；你第五遍重读论文，还是在找同样的错误；你漏掉了一个公式里的错别字，结果被学生或审稿人发现。
  • 没有人检查数学推导。语法检查器能找到"the"，但发现不了分解定理里符号写反了、回归模型设定错误、复现代码跑出了错误结果。

本工作流解决了上述所有问题。你描述自己想要什么——"把第5讲翻译成 Quarto"、"投稿前帮我审一遍论文"、"分析这个数据集并生成发表级表格"——Claude 负责其余一切：制定方案、实施方案、运行专业审阅器、修复问题、验证质量、汇报结果。就像一个管理整个项目的总承包商。

1.2 Claude Code 的独特之处

Claude Code 在你的电脑上运行，可完整访问你的文件系统、终端和 Git。它可作为 CLI 工具、VS Code 插件或通过 Claude Desktop 使用——能力相同，配置相同，界面不同。这使以下功能成为可能：

能力对你意味着什么
读写你的文件直接对 .tex、.qmd、.R 文件进行精准编辑
执行 shell 命令直接编译 LaTeX、运行 R 脚本、渲染 Quarto
访问 Git 历史在对话中完成 commit、PR、分支管理
持久记忆CLAUDE.md + MEMORY.md 跨会话保存
编排器模式Claude 自主计划、实施、审阅、修复、验证
多 Agent 工作流10 个专业 Agent，负责校对、布局、教学法、代码审阅等
质量门控自动评分——低于 80/100 不允许提交
💡 案例：Emory 大学 Econ 730

本工作流是在构建一门"因果面板数据"博士课程的 6 次以上会话中开发出来的。成果：6 套完整讲座（每套 140+ 张幻灯片），含 Beamer + Quarto 双版本、交互式 Plotly 图表、TikZ 图示和 R 复现脚本——全部由编排器管理，并经 5 个质量维度上的 10 个专业 Agent 审阅。

虽然案例以幻灯片为主，但所有组件——Agent、编排器、质量门控——对研究论文、数据分析和课题申请同样适用。

1.3 整体运作原理

在深入配置之前，有一个关键认知：本工作流大部分是自动运行的。你用自然语言描述想要什么，Claude 自行决定使用哪些工具。

1.3.1 你做什么 vs. 自动发生什么

你做的事自动发生的事
描述你想要什么Claude 选择并运行合适的 Skill
批准方案编排器协调各 Agent
查看最终产出钩子在特定事件（编辑/保存/压缩）时自动触发
准备好时说"commit"规则根据你操作的文件自动加载

1.3.2 示例："明天上课前帮我修好幻灯片"

你："审查我的讲座幻灯片，在明天上课前修好所有问题" ↓ Claude 自动： → 运行 /proofread（语法、错别字、一致性） → 运行 /visual-audit（溢出、布局、间距） → 运行 /pedagogy-review（叙事流、符号清晰度） → 综合发现，生成优先级修复清单 → 应用修复（关键 → 重要 → 次要） → 重新验证所有内容能正常编译 → 对照质量门控评分 ↓ 你看到："完成。修复了 12 处问题。得分：88/100。可以 commit 了吗？" 你："可以" ↓ Claude 执行 /commit（仅在你明确批准后）

1.3.3 你永远不需要直接操作的内容

  • Agents — 由 Skill 调用的专业审阅器，不是你直接调用的
  • Hooks — 在特定事件自动触发，你不需要手动运行
  • Rules — 根据文件路径自动加载，无需手动干预

1.3.4 Skills：你可能会用到的唯一命令

/proofread、/compile-latex 这类 Skill 有两种调用方式：

  1. 主动调用 — 你输入 /proofread MySlides.tex
  2. 自动调用 — Claude 在判断相关时自动调用

大多数时候，你只需描述目标，Claude 处理其余。主动调用 Skill 是在你想要精准控制时使用的。

⚡ 核心结论

你说话，Claude 编排。10 个 Agent、22 个 Skill、18 条 Rule 存在的意义，就是让你不用去想它们。描述目标，批准方案，让系统运转。

📌 第一天不需要掌握全部

本指南描述的是完整系统——10 个 Agent、22 个 Skill、18 条 Rule。那是上限，不是门槛。只需从 CLAUDE.md 和 2–3 个 Skill 开始（/compile-latex、/proofread、/commit）。随着使用深入，再按需添加 Rule 和 Agent。模板设计为渐进式采纳：Fork 后填写占位符，然后开始工作。其余的在你准备好时随时可用。

第三章系统实战演示

配置完成后，系统实际能做什么？本章介绍让工作流强大的三个核心机制：专业化 Agent、对抗性质量保证、以及自动质量评分。

3.1 为何专业化 Agent 优于通用做法

假设你要校对一套 140 张幻灯片的讲座。你可以问 Claude：

"帮我审阅这些幻灯片的语法、布局、数学正确性、代码质量和教学流。"

Claude 会大致浏览并抓到一些问题。但它会漏掉：

  • 第 42 张幻灯片中下标从 \(m_t^{d=0}\) 变成了 \(m_t^0\)
  • TikZ 图中两个标签在演示分辨率下重叠
  • R 脚本使用 k=10 个协变量，但幻灯片写的是 k=5

现在对比专业化 Agent：

Agent专注领域能发现什么
proofreader仅语法"principle" 写成了 "principal"
slide-auditor仅布局第 37 张文字溢出
pedagogy-reviewer仅教学流定理 3.1 之前缺少铺垫句
r-reviewer仅代码缺少 set.seed()
domain-reviewer实质内容幻灯片说 10,000 次 MC 模拟，代码只跑了 1,000 次

每个 Agent 读取同一文件，但以全部注意力检查不同维度。/slide-excellence 技能并行运行所有这些 Agent。

3.2 对抗性模式：评审者 + 修复者

系统中最强大的单一模式是对抗性质量保证循环：

+------------------+ | quarto-critic | "我发现 12 处问题。3 处严重。" | （只读） | +--------+---------+ | +----v----+ | 裁决 | +----+----+ / \ 批准 需要改进 | | 完成 +----v---------+ | quarto-fixer | "已修复 12/12 处问题。" | （只写） | +----+---------+ | +----v----------+ | quarto-critic | "重审：仍有 2 处问题。" | （第 2 轮） | +----+----------+ | ... （最多 5 轮）

为什么有效：评审者不能修改文件（只读），所以它没有动机淡化问题。修复者不能给自己打分（评审者会重新审核）。这防止了 Claude 对自己的作品说"看起来不错"这种常见失效模式。

💡 真实案例

在 Econ 730 第 6 讲中，评审者发现 Quarto 版本在 Beamer 版本有完整 Hajek 权重公式的地方使用了 \cdots（占位符）。修复者替换了它。重审时，评审者又发现了 8 处结果模型中缺少 (X) 参数的实例。经过 4 轮后，Quarto 幻灯片与 Beamer 源文件完全一致。

3.3 编排器：自动协调 Agent

单个 Agent 是专家。/slide-excellence、/qa-quarto 等 Skill 为特定任务协调几个 Agent。但在日常工作中，你不应该需要考虑运行哪些 Agent——这是编排器的工作。

编排器协议（.claude/rules/orchestrator-protocol.md）是一条在任何方案获批后自动激活的规则。它实施方案、运行验证器、根据文件类型选择审阅 Agent、应用修复、重新验证，并对照质量门控评分。它循环运行，直到得分达到阈值或达到最大轮数。

你永远不需要手动调用编排器——它是任何非简单任务的默认运行模式。Skill 仍可用于单独使用（如用 /proofread 快速检查语法），但编排器处理完整的生命周期。详见模式 2。

3.4 质量评分系统：80/90/95

每个文件都会得到一个 0 到 100 的质量评分：

分数门控阈值含义操作
80+Commit可以安全保存进度允许 git commit
90+PR准备好部署鼓励 gh pr create
95+卓越出色质量追求的目标
< 80阻止存在严重问题提交前必须修复

3.4.1 分数计算方式

每发现一个问题就扣分：

问题类型扣分为何严重
公式溢出-20数学被截断 = 无法使用
引用失效-15学术诚信问题
公式错别字-10教了错误内容
文字溢出-5内容被截断
标签重叠-5图示无法辨读
符号不一致-3让学生困惑

3.4.2 强制验证

验证协议（.claude/rules/verification-protocol.md）要求 Claude 在报告任务完成前，必须编译、渲染或以其他方式验证每个产出。编排器将此作为其循环中的一个明确步骤（第 2 步：验证）强制执行。这意味着 Claude不能在没有实际检查产出的情况下说"完成"。

⚠️ 不要跳过验证

在 Econ 730 中，验证发现了：部署后标签重叠的未验证 TikZ 图示、Quarto 幻灯片中无法显示的损坏 SVG，以及因缺少截距项而产生静默错误估计的 R 脚本。

第四章核心构建模块

理解配置层级有助于你定制工作流，也能在出问题时快速调试。Claude Code 的强大来自五个协同工作的配置层——可以将它们视为学术项目的操作系统。

4.1 CLAUDE.md — 项目"宪法"

CLAUDE.md 是最重要的文件。Claude 在每次会话开始时读取它。但有一个关键认知：Claude 可靠地遵守约 100–150 条自定义指令。你的系统提示已使用约 50 条，留给你的项目约 100–150 条。CLAUDE.md 和始终加载的规则共用这个预算。

这意味着 CLAUDE.md 应该是一部精简的宪法——简短的指令和指针，而非全面的文档。目标约 120 行：

  • 核心原则 — 4–5 条（计划优先、验证在后、质量门控、LEARN 标签）
  • 文件夹结构 — 所有内容放在哪里
  • 命令 — 编译、部署、关键工具
  • 定制化表格 — Beamer 环境、CSS 类
  • 当前状态 — 已完成什么、进行中什么
  • Skill 快速参考 — 可用斜杠命令的表格

4.5.2 多模型策略：成本 vs. 质量

并非所有 Agent 都需要相同的模型。每个 Agent 文件在 YAML 前置信息中有一个 model: 字段：

任务类型推荐模型原因示例
复杂翻译model: opus需要深入理解两种格式beamer-translator, quarto-critic

原则：对需要同时在脑中持有两个大文档的任务使用 Opus（翻译、对抗性比较）。对有清晰、有限范围的任务使用 Sonnet（修复这 12 处问题、检查这个 R 脚本）。让其他的继承主会话模型。

💰 节省成本

如果你按 Agent 配置模型，一次典型的 Beamer 转 Quarto 任务会用 Opus 运行评审者（2–4 轮），用 Sonnet 运行修复者（相同轮数）。与全部使用 Opus 相比，这能节省约 40–60% 的费用，而修复步骤的质量不会有任何损失。

4.7.1 Plans（方案）— 抗压缩的任务记忆

MEMORY.md 存储长期项目事实，而方案存储任务特定策略。每个非简单任务的方案都以时间戳保存到 quality_reports/plans/。这意味着：

  • 方案在自动压缩中存活（它们在磁盘上，不只在上下文中）
  • 方案在会话边界间存活（在未来任何会话中可读）
  • 方案创建了设计决策的审计轨迹

4.7.2 会话日志 — 记录"为什么"而非"做了什么"

Git commit 记录什么改变了，但不记录为什么。会话日志填补了这一空白。Claude 在三个时间点写入 quality_reports/session_logs/：方案批准后、实施过程中（随着决策发生），以及会话结束时。

由于单靠指令是脆弱的（Claude 在长会话中会遗忘），Stop 钩子（.claude/hooks/log-reminder.py）在每次响应后触发。它跟踪自上次更新会话日志以来经过了多少次响应。超过阈值后，它会阻止 Claude 停止，直到日志是最新的。

📝 为什么会话日志很重要

Git 记录做了什么；会话日志记录为什么。一条 commit 信息说"更新第 5 讲 TikZ 图示"。一条会话日志说"重新设计了 TWFE 分解图，因为 DA 挑战揭示学生无法追踪从权重到偏差的路径。考虑过表格格式，但选择了流程图因为它显示了方向性。"

增量记录是关键。一个 4 小时的会话如果只在开始和结束时记录，会丢失中间的所有内容。随着决策发生立即追加，意味着自动压缩永远无法清除它们——它们已经在磁盘上了。

第五章工作流模式

前两个模式是元模式——它们控制每个任务的流程。先掌握这两个，其他具体工作流就更容易理解了。

模式 1：计划优先开发

计划优先模式确保非简单任务以思考开始，而非直接动手。

为什么规划很重要

AI 辅助开发中最常见的失败模式不是糟糕的代码——而是解决了错误的问题，或以脆弱的顺序解决了正确的问题。计划优先开发在触碰任何文件之前强制执行一个明确的设计步骤。

协议流程

非简单任务到来
  |
  +-- 第 1 步：Claude 进入规划模式（自动，或说"先规划一下"）
  +-- 第 2 步：起草方案（方法、文件、验证）
  +-- 第 3 步：保存到 quality_reports/plans/YYYY-MM-DD_描述.md
  +-- 第 4 步：向用户呈现方案
  +-- 第 5 步：用户批准（或修改）
  +-- 第 6 步：保存初始会话日志（趁上下文新鲜时捕获）
  +-- 第 7 步：编排器接管（见模式 2）
  +-- 第 8 步：更新会话日志 + 方案状态改为 COMPLETED
💡 每周回顾

对于多项目的学者，每周开始时请 Claude 读取过去一周的所有会话日志，综合生成状态报告，列出优先级和待解决问题。会话日志基础设施已经捕获了你需要的一切——每周回顾只是一个综合提示："读取本周所有会话日志。总结：完成了什么、什么受阻、我下一步应该优先处理什么？"

模式 2：承包商模式（编排器）

方案批准后，编排器接管。它是模式 1 的自然延续：方案说做什么，编排器自主处理怎么做。

心智模型

将编排器想象成总承包商。你是甲方。你描述想要什么。计划优先协议是蓝图阶段。一旦你批准蓝图，承包商接管：雇用合适的专家（Agent）、检查他们的工作（验证）、让他们回去修复问题（审阅-修复循环），只有在工程通过验收（质量门控）时才来找你。

编排器循环

用户："将第 5 讲转换为 Quarto" | |-- 计划优先（模式 1）：起草方案，保存到磁盘，获得批准 | |-- 用户："已批准" | +-- 编排器激活： | 第 1 步：实施 | 执行方案步骤（创建 QMD、翻译内容等） | 第 2 步：验证 | 运行验证器：渲染 Quarto，检查 HTML 产出 | 如果渲染失败 -> 修复 -> 重新渲染 | 第 3 步：审阅（根据文件类型选择 Agent） | +--- proofreader --------+ | +--- slide-auditor ------+ （并行） | +--- pedagogy-reviewer --+ | +--- quarto-critic ------+ （需要其他 Agent 先完成） | 第 4 步：修复 | 应用修复：严重 -> 重要 -> 次要 | 对于 quarto-critic 的问题：调用 quarto-fixer | 第 5 步：重新验证 | 再次渲染，确认修复是干净的 | 第 6 步：评分 | 应用质量门控评分标准 | +-- 分数 >= 80？ 是 -> 向用户呈现摘要 否 -> 循环回第 3 步（最多 5 轮）

"直接做"模式

有时你不想批准最终结果——你只想要它完成：

"将第 5 讲转换为 Quarto。直接做。"

在这种模式下，编排器仍然运行完整的验证-审阅-修复循环（质量是不可妥协的），但跳过最终批准暂停，如果分数 80 分及以上则自动 commit。它仍然呈现摘要，让你看到做了什么。

💡 何时使用 Skill vs. 编排器

编排器（自动）："将第 5 讲转换为 Quarto" — 编排器决定调用哪些 Agent。

Skill（明确）："/qa-quarto Lecture5" — 你特别想要对抗性评审者-修复者循环，不需要其他的。

两者都有效。编排器是"我信任你，处理它"的路径。Skill 是"我确切知道我想要什么"的路径。

模式 4：Beamer 转 Quarto

转换保留所有内容，同时适配格式，将 TikZ 转换为 SVG，将 ggplot 图转换为交互式 Plotly 图：

模式 5：复现优先编码

当处理有复现包的论文时：

第 1 阶段：清点原始代码
  +-- 记录"黄金标准"数字（表 X，列 Y = Z.ZZ）

第 2 阶段：翻译（如 Stata -> R）
  +-- 完全匹配原始规格（相同协变量、相同聚类）

第 3 阶段：验证匹配
  +-- 比较每个目标：论文值 vs. 我们的值
  +-- 容忍度：估计值 < 0.01，标准误 < 0.05
  +-- 如果不匹配：停止。在继续之前调查。

第 4 阶段：只有这之后才扩展
  +-- 新的估计器、新的规格、课程特定图表
⚠️ 永远不要跳过复现

在一门课程中，我们发现一个广泛使用的 R 包由于一个微妙的规格问题静默地产生了错误的估计值。这个 bug 在不同脚本中被发现了 3 次。没有复现优先协议，这些错误数字就会教给博士生了。

自我提升有两个层次：通过 [LEARN] 标签进行快速纠错，以及通过 /learn 进行完整技能提取。

7.1 快速纠错：[LEARN] 标签

每条纠错都被标记以供将来在 MEMORY.md 中参考：

模式 14：幻灯片的修辞学

slide-auditor 检查技术质量（溢出、间距）。pedagogy-reviewer 检查教学质量（符号密度、先决条件）。两者都不处理修辞质量——幻灯片是否有说服力、论述是否流畅、美感是否服务于功能。

三大法则

法则一：美就是功能。漂亮的幻灯片不是装饰过度的幻灯片。美感是让清晰度变得可见。每个元素都有存在的理由。没有任何东西会分散对核心内容的注意力。"没有功能的装饰就是噪音。"

法则二：认知负荷是敌人。每张幻灯片一个想法。一个。这不是指导原则——这是法则。观众的工作记忆有限。每个不必要的词、数据点或"以防万一"的内容都在抢夺传递真实信息所需的带宽。

法则三：幻灯片服务于口头表达。"如果你的幻灯片不需要你说话就能被理解，你写的是一份文档，却把它叫做演示。"幻灯片是口头表达的视觉锚点——注意力的焦点，记忆的钩子。

边际效益 / 边际成本等价

本框架最独特的贡献——将边际分析应用于幻灯片设计：

最优修辞使所有幻灯片的边际效益与边际成本之比相等：MB₁/MC₁ = MB₂/MC₂ = … = MBₙ/MCₙ

实践含义：

  • 过载的幻灯片（MB/MC 太低）：文字延伸到页脚、相互竞争的想法、观众放弃
  • 欠载的幻灯片（MB/MC 太高）：浪费机会，注意力被捕获但未被利用
  • 目标是流畅——整个演示保持一致的认知负荷

7.5 6 次以上迭代会话的心得

  1. 保持 CLAUDE.md 在 150 行以内。Claude 可靠地遵守约 150 条指令。400 行的 CLAUDE.md 意味着规则会被静默忽略。对详细标准使用路径限定规则。
  2. 增量式添加规则。不要试图预先写好所有规则。发现模式时再添加。使用 paths: 前置信息使其只在相关时加载。
  3. 使用 [LEARN] 格式。每条纠错都被标记并持久化到 MEMORY.md。这能防止跨会话重复错误。
  4. 信任对抗性模式。评审者-修复者循环会发现你发现不了的问题。让它运行。
  5. 验证一切。验证规则的存在是有原因的。永远不要跳过编译或渲染检查。
  6. 会话日志很重要。记录设计决策，而不只是改变了什么。未来的你会感谢现在的你。
  7. 魔鬼代言人要早用。在你已经在摇摇欲坠的基础上构建了 50 张幻灯片之前，先挑战幻灯片结构。
  8. 渐进式披露。从 CLAUDE.md + 2–3 条规则开始。随着工作流成熟再添加更多。新手不应该第一天就面对 18 条规则。
  9. 用 CLAUDE.local.md 做个人覆盖。此文件自动被 gitignore，与 CLAUDE.md 一起加载。在这里放机器特定的路径、个人偏好和本地工具版本——它们不会污染共享仓库。

8.5 故障排除

8.5.1 LaTeX 无法编译

症状：xelatex 报错或缺少包。

修复：检查是否安装了 XeLaTeX：which xelatex；确保 TEXINPUTS 包含 Preambles/（/compile-latex Skill 处理此事）；缺少包？通过 TeX Live 安装：tlmgr install [包名]

8.5.2 Quarto 无法渲染

症状：quarto render 失败或产生损坏的 HTML。

修复：检查 Quarto 版本：quarto --version（需要 1.3+）；检查 YAML 前置信息中的语法错误；缺少 TikZ SVG？先运行 /extract-tikz

8.5.3 钩子不触发

症状：没有上下文警告，没有验证提醒。

修复：检查钩子是否已配置：cat .claude/settings.json | grep hooks；确保 Python 3 可用：which python3；检查钩子文件权限：ls -la .claude/hooks/

8.5.4 Claude 忽略规则

症状：Claude 不遵循 .claude/rules/ 中的约定。

修复：规则使用 paths: 前置信息——检查路径是否与你的文件匹配；规则太多？Claude 可靠遵守约 150 条指令，合并规则；尝试："读取 .claude/rules/[规则].md 并在此任务中遵循它"

8.5.5 压缩后上下文丢失

症状：Claude 忘记了你在处理什么。

修复：指引 Claude 找到方案；检查会话日志；post-compact-restore.py 钩子应该自动打印恢复信息

8.5.6 质量分数太低

症状：分数卡在 80 以下，无法 commit。

修复：运行 /slide-excellence 获取详细问题分解；首先修复严重问题（每个扣 10–20 分）；问 Claude："剩余的严重问题是什么？"

8.5.7 Skills 没有自动调用

症状：描述任务时 Claude 不使用 Skill。

修复：在请求中明确一些："审阅我的幻灯片的语法和布局问题"；检查 Skill 是否启用了自动调用；Skill 描述帮助 Claude 知道何时使用它——检查描述是否清晰
